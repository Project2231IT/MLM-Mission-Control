## Playback Fix (2026-03-25 morning)
- **Root cause:** I changed TVT/onvif streams from `#video=copy` to `#video=h264` — forced CPU transcode of H265→H264 at 1080p25fps, eating 146% CPU, load avg 7+ on 12-core VM
- **Fix:** Restored `#video=copy` passthrough, switched VideoPlayer to WebRTC (Chrome handles H.265 natively via WebRTC since v107)
- Added TVT playback RTSP URLs (date/time/timelen params) — was completely missing before
- WebRTC player with MP4 fallback deployed
- **Lesson:** NEVER transcode H.265 on this VM. Always use `#video=copy` for TVT/onvif and let WebRTC handle codec negotiation. Only Uniview needs `#video=h264` because their playback has different RTSP format.
- Stream API now waits for go2rtc to have data before returning URL (poll loop up to 8s)
- Jake was frustrated — multiple failed attempts before finding the real issue
